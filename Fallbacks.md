# Fallback 方案说明文档

本文档汇总了项目中为提升鲁棒性而设计的各类降级（Fallback）方案，确保在依赖缺失、硬件环境不满足等异常情况下，程序仍能继续执行并保证核心功能可用。

## 1. 重复生图跳过策略
- **核心目标**：避免重复消耗生图额度，提升执行效率
- **实现逻辑**：在生成 `output/figure.png` 前增加文件存在性校验
- **行为规则**：若目标文件已存在，则直接复用该文件，跳过生图流程；仅当文件不存在时，才执行生图逻辑

## 2. SAM3 EDT 计算回退
- **触发条件**：Triton 库不可用（导入失败/调用报错）
- **降级逻辑**：从 Triton 实现的 EDT（Euclidean Distance Transform）算法，回退到 OpenCV 提供的 distance transform 实现
- **保障效果**：EDT 核心计算功能不受影响，仅可能存在性能差异

## 3. BPE 资源路径回退
- **触发条件**：通过 `pkg_resources` 无法获取 BPE（Byte Pair Encoding）相关文件
- **降级逻辑**：改用基于 `__file__` 的本地相对路径查找 BPE 文件
- **实现价值**：避免因包资源加载机制异常导致的 BPE 功能不可用

## 4. CUDA 硬编码回退
- **优化前**：代码中直接写死设备为 `"cuda"`
- **优化后**：通过 `torch.cuda.is_available()` 动态判断
  - CUDA 可用时：使用 GPU 执行
  - CUDA 不可用时：自动回退到 CPU 执行
- **核心收益**：程序可在无 GPU 环境下正常运行

## 5. CPU 环境适配优化
- **优化点**：移除仅适用于 CUDA 环境的内存优化逻辑
- **具体操作**：删除 `pin_memory()`、`non_blocking=True` 等 CUDA 专属参数
- **优化原因**：这些参数在 CPU 环境下无意义，且可能导致运行报错

## 6. 关键警告容忍机制
- **容忍的警告类型**：
  - `autocast cuda not available`（CUDA 自动混合精度不可用）
  - `Triton is not available`（Triton 库不可用）
- **处理逻辑**：捕获警告并记录，主流程不中断
- **最终效果**：程序仍能执行到生成 `final.svg` 的最终步骤

## 7. Web 服务端口占用回退
- **触发条件**：默认端口 `8000` 已被占用
- **降级逻辑**：启动阶段按顺序探测后续可用端口，并自动切换到首个可监听端口
- **已验证结果**：本地实际验证中，`8000` 被占用时服务成功回退到 `8001` 并正常启动
- **当前约束**：此策略解决的是端口占用问题，不负责主动终止其他历史服务进程
